Performance & Scalability Architecture

Design for production: caching, query optimization, async processing, and scale

Cache Invalidation Strategy:
  • Metadata cache: Invalidate on entity schema change (webhook from migration)
  • Permission cache: Invalidate on role/permission change (event emitted)
  • Computed field cache: Invalidate on source field update (in post-hook)
  • Query cache: Invalidate on any write to entity (blanket invalidation for safety)
  • Lazy TTL: If cache miss, fetch from DB and repopulate immediately
2. Query Optimization
PRINCIPLE: Push filtering and sorting to database

Problem: Fetch 10,000 tickets, filter in-memory → slow
Solution: Push permissions + filters to SQL WHERE clause

// Runtime builds optimal query:

const user = { id: 'user123', role: 'support_agent' }
const filter = { status: 'open', priority: 'high' }
const sort = { createdAt: 'desc' }

// 1. Load permission policy from cache
const policy = cache.get('policy:ticket:user123')
// Rule: "assignedTo = user.id OR role = 'manager'"

// 2. Build WHERE clause
const where = [
  // Permission filter (ABAC)
  { OR: [
      { assignedTo: user.id },
      { role: 'manager' }  // This is on the User relation
    ]
  },
  // User filters
  { status: 'open' },
  { priority: 'high' },
  // Soft delete filter (automatic)
  { deletedAt: null }
]

// 3. Build SELECT (only requested columns)
const select = {
  id: true,
  title: true,
  status: true,
  priority: true,
  assignedTo: { select: { id: true, name: true } },
  // Don't fetch: description, resolution, etc unless needed
}

// 4. Execute optimized query
const result = await drizzle
  .select(select)
  .from(tickets)
  .where(where)
  .orderBy(desc(tickets.createdAt))
  .limit(20)
  .offset(0)
  .execute()

// Query executed:
// SELECT id, title, status, priority, assignedTo.id, assignedTo.name
// FROM tickets
// LEFT JOIN users ON assignedTo = users.id
// WHERE (assignedTo = 'user123' OR users.role = 'manager')
//   AND status = 'open'
//   AND priority = 'high'
//   AND deletedAt IS NULL
// ORDER BY createdAt DESC
// LIMIT 20

// Result: 1 query, not 10,000
// Execution time: ~50ms
// Memory: ~100KB instead of 10MB

DATABASE INDEXES (Automatic):

For each entity, create indexes on:
1. Soft delete: (deletedAt) — speeds up soft delete filter
2. Relationships: (assignedTo) — speeds up permission joins
3. Search fields: (title, description) FULLTEXT — enables search
4. Sort fields: (createdAt, updatedAt) — speeds up sorting
5. Filter fields: (status, priority, category) — speeds up filtering
6. Composite: (assignedTo, status) — common query pattern

INDEX: tickets_assignedTo_status
  → Speeds up: "WHERE assignedTo = ? AND status = ?"

INDEX: tickets_createdAt_deletedAt
  → Speeds up: "WHERE deletedAt IS NULL ORDER BY createdAt"

──────────────────────────────────────────────────────────────────

PAGINATION (Always):

Never return more than 100 records per request
├─ Default: 20 records
├─ Max: 100 records
└─ Cursor-based pagination for large datasets

GET /api/tickets?limit=20&offset=0
GET /api/tickets?limit=20&cursor=ticket_456

──────────────────────────────────────────────────────────────────

RELATED DATA (Lazy Loading):

Don't fetch relations by default
├─ Include: GET /api/tickets?include=assignedTo,customer
├─ Depth limit: max 2 levels (ticket → customer → account)
└─ Size limit: related data < 1MB per request

──────────────────────────────────────────────────────────────────

SEARCH OPTIMIZATION (Full-Text):

For large text fields, use database full-text search
├─ FULLTEXT INDEX on title, description
├─ Query: SELECT * FROM tickets WHERE MATCH(title) AGAINST(?)
├─ Result: 1-2ms for 1M+ records
└─ No need for Elasticsearch (unless search > 5M records)
3. Async Processing (Event-Driven)
PRINCIPLE: Don't block on I/O; queue and process async

Synchronous Operations (< 50ms):
├─ CRUD: Insert/update/delete record in database
├─ Permission check: Query Redis cache
├─ Validation: In-memory schema check
├─ Hooks: beforeCreate, beforeUpdate (in-process)
└─ Response: Return immediately to client

Asynchronous Operations (background):
├─ Workflows (send emails, Slack, webhooks)
├─ Agents (scheduled jobs, escalations)
├─ Cache invalidation (queue all invalidations)
├─ Audit logging (write to audit table async)
├─ Real-time updates (WebSocket broadcast)

EVENT BUS (BullMQ):

1. Record created → Emit: { type: 'ticket.created', data: {...} }
2. Event handler queues jobs:
   ├─ Job: 'email.send' { workflowId: 'ticket.notify_customer', ... }
   ├─ Job: 'slack.notify' { channel: '#support', message: '...' }
   └─ Job: 'webhook.trigger' { url: '...', payload: {...} }
3. Worker processes queue:
   ├─ Max concurrency: 10 workers per job type
   ├─ Retry: 3 attempts with exponential backoff
   ├─ Timeout: 30 seconds per job
   └─ DLQ: Failed jobs moved to dead-letter queue after 3 retries
4. Audit trail recorded (async):
   ├─ Job status: pending → processing → completed/failed
   ├─ Duration: when started, when finished
   └─ Error: if failed, error message stored

Queue Structure:
┌─────────────────────────────────────────┐
│ BullMQ Job Queue (Redis)                │
├─────────────────────────────────────────┤
│ waiting     [job1, job2, job3, ...]     │
│ active      [job4, job5]                │
│ completed   [job6, job7, ...]           │
│ failed      [job8]                      │
│ delayed     [job9]                      │
└─────────────────────────────────────────┘

Worker Pool:
┌───────────────────────────────────────────────┐
│ Worker 1: Processing job4                     │
│ Worker 2: Processing job5                     │
│ Worker 3: Idle (waiting for jobs)             │
│ Worker 4: Idle                                │
│ ...                                           │
│ Worker 10: Idle                               │
└───────────────────────────────────────────────┘

Scaling:
├─ Jobs queuing: Linear (1 Redis write per job)
├─ Job processing: Parallelizable (add more workers)
├─ Queue monitoring: Built-in UI (BullMQ Board)
└─ Throughput: 1000 jobs/min with 10 workers
4. Bottleneck Analysis & Solutions
BOTTLENECK: Database Query (Slow JOIN)
Problem: JOIN on permissions table for every request
Solution:
  ├─ Cache compiled permission policies in Redis
  ├─ Build WHERE clause in-app (not in DB)
  ├─ Use database indexes on foreign keys
  └─ Result: 50ms → 5ms per query

BOTTLENECK: Computed Fields (Expensive Calculation)
Problem: Calculating "daysOpen" for every record
Solution:
  ├─ Cache computed values in Redis (5 min TTL)
  ├─ Only recalculate on record update
  ├─ Use database computed columns if possible
  └─ Result: No database hit, just cache read (1ms)

BOTTLENECK: Webhook Timeout (External Service Slow)
Problem: Workflow waiting for webhook response (5s timeout)
Solution:
  ├─ Make workflows async by default (don't wait)
  ├─ Retry with exponential backoff (BullMQ)
  ├─ Use fire-and-forget for non-critical webhooks
  └─ Result: Response returned immediately (50ms)

BOTTLENECK: Large Result Sets (Fetch 10k records)
Problem: Memory overflow, slow serialization
Solution:
  ├─ Enforce pagination (max 100 per request)
  ├─ Streaming responses (JSON Lines format)
  ├─ Limit to searchable fields only
  └─ Result: Always < 1MB per response

BOTTLENECK: Concurrent Writes (Race Condition)
Problem: Two users updating same record simultaneously
Solution:
  ├─ Optimistic locking: Check version before update
  ├─ Last-write-wins strategy (with audit trail)
  ├─ Database constraint: UNIQUE(id, version)
  └─ Result: Consistent state, no data loss

BOTTLENECK: Permission Check (ABAC Expression)
Problem: Complex ABAC rule evaluation
Solution:
  ├─ Pre-compile rules to efficient bytecode
  ├─ Short-circuit evaluation (stop at first false)
  ├─ Cache evaluation results (policy:{userId}:{action})
  └─ Result: 100µs per check (in memory)

Key Performance Principles:
  • Cache aggressively: Metadata, permissions, computed fields all cached in Redis
  • Push filtering down: Permissions and filters evaluated in SQL, not in-app
  • Async by default: Workflows, agents, notifications queued, never blocking
  • Pagination always: Never return > 100 records; use cursor-based for large datasets
  • Index strategically: Indexes on soft delete, relationships, search, sort, filters
  • Monitor actively: Built-in performance metrics, slow query logs, queue monitoring
