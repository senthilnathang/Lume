Integration Points

  • Entity → Permission: Every CRUD operation checks permission policy before execution
  • Entity → Workflow: Record creation/update triggers workflows defined in entity.workflows
  • Entity → View: Entity fields auto-generate form/grid/kanban views; views can be customized
  • Entity → Agent: Record updates trigger agents that react to conditions (e.g., status change)
  • Workflow → Permission: Workflow actions must respect permissions (e.g., can't reassign without permission)
  • View → Permission: Views hide columns/actions user doesn't have permission for
  • Agent → Workflow: Agents can trigger workflows as reaction to conditions
Detailed Integration Examples
──────────────────────────────────────────────────────────────────
EXAMPLE 1: TICKET LIFECYCLE (All systems integrated)
──────────────────────────────────────────────────────────────────

1. USER CREATES TICKET
   ├─ Request: POST /api/tickets { title: "...", description: "..." }
   ├─ Permission: Check if user.role has 'ticket.create' ✓
   ├─ Hook: beforeCreate executes, auto-assigns to available rep
   ├─ Workflow: 'notify-on-create' triggered → send email
   ├─ View: Form validates fields per entity definition
   └─ Result: Ticket created, email sent, rep notified

2. AGENT REACTS: Check overdue tickets (runs every 5 mins)
   ├─ Trigger: Record where createdAt < now - 24h AND status != 'closed'
   ├─ Action: Emit 'ticket.overdue' event
   ├─ Workflow: 'escalate-overdue' triggered
   ├─ Side effect: Update priority, notify manager
   └─ View: Dashboard red-flags overdue tickets (via permission scope)

3. MANAGER UPDATES TICKET
   ├─ Request: PUT /api/tickets/123 { status: 'closed' }
   ├─ Permission: Check if user.role can 'ticket.update' ✓
   ├─ Hook: beforeUpdate validates workflow state (can't close if assigned)
   ├─ Workflow: 'notify-on-resolve' triggered → send customer email
   ├─ Agent: Reaction triggered (status=closed) → update rep's stats
   ├─ View: Status field changes, form shows resolution notes required
   └─ Result: Ticket closed, customer notified, stats updated

4. CUSTOMER VIEWS TICKET (Public Portal)
   ├─ Request: GET /api/tickets/123?view=public
   ├─ Permission: Filter by scope (customer sees only their tickets)
   ├─ View: Load 'public' view definition (limited fields)
   ├─ Computed fields: 'daysOpen' = now - createdAt
   ├─ Side effect: Update 'last_viewed_by_customer' timestamp
   └─ Result: Public view rendered with allowed columns/actions

EXAMPLE 2: PERMISSION-DRIVEN BEHAVIOR
──────────────────────────────────────────────────────────────────

Entity.permission: {
  'ticket.read': {
    rule: "user.role='support_agent' OR createdBy=user.id",
    scope: "return only records where condition is true"
  },
  'ticket.update': {
    rule: "user.role='manager' OR assignedTo=user.id",
    fieldLevel: {
      'status': "user.role='manager' only",
      'priority': "user.role='manager' only",
      'notes': "anyone"
    }
  },
  'ticket.delete': {
    rule: "user.role='admin' only"
  }
}

When user loads tickets grid:
├─ Query filtered: WHERE (user.role='support_agent' OR createdBy=user.id)
├─ Columns shown: title, description, status, notes (hidden: internal_notes)
├─ Actions shown: Edit, Reply (hidden: Delete, Reassign)
└─ Inline edit: Only 'notes' editable (others locked)

When user clicks Update:
├─ Permission check: Can user modify this field?
├─ If no: Field locked with "You don't have permission" tooltip
├─ If yes: Field editable, validation runs, update succeeds

EXAMPLE 3: VIEW AUTO-GENERATION
──────────────────────────────────────────────────────────────────

Entity fields automatically generate views:

Input Entity Definition:
├─ title (text, required)
├─ description (text)
├─ status (select: ['open', 'in_progress', 'closed'])
├─ assignedTo (relation: User)
└─ createdAt (date)

Auto-Generated Views:
├─ Form View
│  └─ Layout: title → description → status → assignedTo
│     Validation: title required
│     Fields locked per permissions
│
├─ Table/Grid View
│  └─ Columns: title, status, assignedTo, createdAt
│     Sorting: any column
│     Filtering: status, assignedTo
│     Inline edit: enabled for permitted fields
│
├─ Kanban View
│  └─ Swim lanes: status (open → in_progress → closed)
│     Cards: title, assignedTo
│     Drag-to-update: change status by moving card
│
└─ Custom Views (user-defined)
   └─ Kanban with different grouping, custom filters
