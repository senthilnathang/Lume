Runtime Execution Flow

Step-by-step: How a request flows through the pipeline

Example: Creating a Ticket Record
CLIENT REQUEST:
┌─ POST /api/tickets
├─ body: { title: "Login broken", description: "Users can't log in" }
└─ headers: { Authorization: "Bearer token...", X-Org-Id: "org123" }

        │
        ▼

STEP 1: LOAD EXECUTION CONTEXT
├─ Extract user from JWT token → userId: "user123"
├─ Extract org from header → orgId: "org123"
├─ Load user roles/permissions from database
└─ Result: ExecutionContext { userId, orgId, roles, permissions }

        │
        ▼

STEP 2: FETCH METADATA
├─ Registry.getEntityMetadata('ticket')
├─ Returns: EntityDefinition {
│   slug: 'ticket',
│   orm: 'drizzle',
│   tableName: 'tickets',
│   fields: [...],
│   hooks: {...},
│   workflows: {...}
│ }
└─ All config loaded into memory (cached)

        │
        ▼

STEP 3: BUILD OPERATION REQUEST
├─ OperationRequest {
│   context: ExecutionContext,
│   entity: 'ticket',
│   action: 'create',
│   data: { title: "...", description: "..." },
│   options: { includeRelations: [] }
│ }
└─ Normalized request ready for pipeline

        │
        ▼

STEP 4: RUN INTERCEPTOR PIPELINE (8 stages)
│
├─ [10] AuthInterceptor
│   ├─ Check: Is user authenticated?
│   ├─ Result: ✓ Allowed (user token valid)
│   └─ Pass to next interceptor
│
├─ [20] PermissionInterceptor (RBAC/ABAC)
│   ├─ Policy: "Can user 'user123' CREATE tickets?"
│   ├─ Check: role='support_agent' has permission 'ticket.create'
│   ├─ Result: ✓ Allowed
│   └─ Store in context.permissions
│
├─ [30] SchemaValidator
│   ├─ Validate: title (required, text) → "Login broken" ✓
│   ├─ Validate: description (text) → "Users can't log in" ✓
│   ├─ Result: ✓ Valid
│   └─ Errors: none
│
├─ [40] PreHooks
│   ├─ Execute: entity.hooks.beforeCreate()
│   ├─ Custom logic: Call ticket.service.assignToAvailableRep()
│   ├─ Mutates record: Add assignedTo: "rep456"
│   └─ Result: Transformed record ready to persist
│
├─ [50] OrmSelector
│   ├─ Check: entity.orm = 'drizzle'
│   ├─ Load: DrizzleAdapter instance
│   └─ Result: Ready to execute
│
├─ [60] QueryExecutor
│   ├─ ORM: drizzle.insert(tickets).values({...})
│   ├─ Database: INSERT INTO tickets (title, description, assignedTo) VALUES (...)
│   ├─ Result: { id: "ticket_123", title: "...", assignedTo: "rep456", createdAt: now }
│   └─ Persisted record returned
│
├─ [70] PostHooks
│   ├─ Execute: entity.hooks.afterCreate()
│   ├─ Custom logic: Update rep's workload, send notification
│   └─ Result: Side effects completed
│
└─ [80] EventEmitter
    ├─ Emit event: { type: 'ticket.created', data: { id: "ticket_123", ... } }
    ├─ Event bus (BullMQ): Queue async work
    ├─ Triggers: ['auto-assign', 'send-notification'] workflows
    └─ Result: Events queued for async processing

        │
        ▼

STEP 5: BUILD RESPONSE
├─ OperationResult {
│   success: true,
│   data: { id: "ticket_123", title: "...", assignedTo: "rep456" },
│   metadata: {
│     executedAt: "2024-04-30T10:30:00Z",
│     duration: "45ms",
│     permissions: [{ check: 'ticket.create', allowed: true }]
│   }
│ }
└─ Return to client

        │
        ▼

CLIENT RECEIVES:
├─ 200 OK
├─ { success: true, data: {...} }
└─ UI updates immediately

        │
        ▼

STEP 6: ASYNC WORKFLOWS (Background)
├─ Event bus processes queued events
├─ Workflow 'send-notification':
│   ├─ Load email template
│   ├─ Render with record data
│   ├─ Send email to customer
│   └─ Mark complete
├─ Workflow 'auto-assign':
│   ├─ Check escalation rules
│   ├─ Update ticket priority if needed
│   └─ Notify assigned rep
└─ Workflows complete independently, don't block response

Key Points

  • Synchronous response: Client gets result in ~50ms (steps 1-5)
  • Asynchronous workflows: Notifications, escalations run in background (step 6)
  • Single flow: Every operation (create, update, delete, read) follows same 8-stage pipeline
  • Metadata-driven: All behavior (hooks, workflows, permissions) defined in EntityDefinition
  • Extensible: Custom interceptors can be inserted at any stage
  • Auditable: Every decision (auth check, permission check) logged in metadata
Error Handling Path
If any interceptor rejects the request:

├─ [20] PermissionInterceptor: User lacks 'ticket.create' permission
│   ├─ Call: interceptor.abort(new PermissionDeniedError(...))
│   └─ Result: Execution stops, jump to error response
│
├─ [30] SchemaValidator: Required field 'title' missing
│   ├─ Call: interceptor.abort(new ValidationError(...))
│   └─ Result: 400 Bad Request with field errors
│
├─ [60] QueryExecutor: Database constraint violation (unique email)
│   ├─ Call: interceptor.abort(new DatabaseError(...))
│   └─ Result: 409 Conflict with specific constraint info
│
└─ Final Response:
    {
      success: false,
      errors: [
        { field: 'email', message: 'Email already exists' }
      ],
      metadata: {
        failedAt: 'SchemaValidator',
        duration: '15ms'
      }
    }
Questions for you:
  • Should metadata (EntityDefinition) be cached in-memory with TTL, or fetched fresh each time?
  • For async workflows: should we support transaction rollback if a workflow fails?
  • Should the response include which interceptors ran and their results?
