Unified Runtime Architecture — Three Approaches

Each balances simplicity, extensibility, and performance differently

A

Layered Execution Model

How it works: Each system (Entity, Workflow, View, Permission) is a standalone layer. Runtime orchestrates by calling each layer in sequence.

Pros: Simple to understand, easy to test each layer independently, loose coupling

Cons: More network overhead (layer-to-layer calls), harder to optimize cross-layer logic, performance hits for complex scenarios

B

Event-Driven Reactive Model

How it works: Central event bus. Entity changes emit events. Workflows, views, permissions react asynchronously via listeners/subscribers.

Pros: Decoupled, scales well, natural fit for async workflows, agents fit naturally

Cons: Harder to reason about execution order, eventual consistency challenges, debugging is harder

C

Metadata Registry + Interceptor Pipeline

How it works: Single source of truth (metadata registry). Every operation passes through an interceptor pipeline (permissions → validation → transformation → execution → side effects).

Pros: Unified flow, cross-cutting concerns handled uniformly, predictable execution, easy to debug, natural fit for NestJS interceptors

Cons: More complex core, single point of failure if not careful, requires careful pipeline design

My recommendation: C with B as a sub-layer. Use the Metadata Registry + Interceptor Pipeline as the core execution model (NestJS-friendly), but add an event bus underneath for async workflows and agent reactions. This gives you the predictability of C with the decoupling benefits of B.
